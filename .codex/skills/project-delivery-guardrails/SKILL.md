# 项目交付护栏

当任务会改动多个模块、需要先定架构再写代码，或用户明确要求“先梳理方案、保证文件位置正确、代码不要越改越乱”时，优先使用本 skill。

## 先做架构梳理再动手

在写代码前先完成这四件事：

- 识别当前需求落在哪一层，不要一上来就补函数或塞新目录。
- 盘点现有模块职责、依赖方向和文件落位，找出重复抽象、错误归属和临时拼接点。
- 先给出目标落位方案：新增文件放哪里、旧文件是否该移动、哪些接口该收敛、哪些概念不该继续扩散。
- 把实施顺序拆成最小闭环，优先做结构正确、长期可维护的方案，而不是最低成本的临时补丁。

如果仓库已经存在稳定模式，先复用既有边界；如果现有边界明显混乱，再提出收敛方案后实现。

## 重点检查项

- 目录是否按领域和职责分层，而不是按一次性需求堆散文件。
- 抽象是否服务于复用和边界，而不是为了“看起来灵活”引入额外跳板层。
- 新逻辑是否放在真正拥有该职责的模块，而不是放在最顺手修改的文件。
- 命名、配置入口、测试入口和文档是否能沿着同一套结构找到。
- 改动是否会把运行时职责重新压回大而杂的模块、脚本或 Lambda。

## 本仓库的架构基线

业务完整链路固定表述为：

```text
S3 -> OCR -> Embedding -> S3 Vectors
```

当前默认运行时是：

```text
source bucket
-> S3 Event Notification
-> SQS ingest queue
-> Ingest Lambda
-> Step Functions Standard
   -> extract workflow lambdas
   -> manifest bucket
   -> SQS embed queue
-> Embed Lambda
-> Embedding provider
-> S3 Vectors
-> SQS DLQ
-> Query Service
```

外部只暴露 `Ingest Lambda`，内部允许拆分成多段 Lambda，但不能把职责压回单个长函数。
本地编排集成层应显式覆盖 `S3`、`DynamoDB`、`SQS`、`Step Functions` 和 `Lambda` 的本地仿真，并保持为 PR 可见门禁。
S3 Vectors 没有本地原生等价实现时，只允许做适配层验证，不要把本地替代品当成语义完全等价实现。
Python 依赖与测试入口默认走 `uv sync --locked`、`uv run`、`uvx`。

## 实施顺序

1. 先说明目标架构和文件落位，再开始改代码。
2. 先处理边界和目录，再补实现细节，避免边改边漂移。
3. 相关测试、文档、skill 和部署口径一起收敛，不要留“以后再补”的结构债。
4. 如果是大任务，优先拆成几段可验证的闭环，而不是一次塞入多个横跨全仓库的半成品。

## 反模式

- 为了兼容当前脏结构，把新逻辑继续塞进历史包袱最重的文件。
- 目录按技术噪音分层，却让业务职责在多个位置横切重复。
- 只补局部函数，不处理明显错误的模块归属和接口边界。
- 先写代码再补架构说明，导致测试、文档、部署命名全部跟着漂移。
- 用一次性的 wrapper、兼容层或转发文件掩盖真正应该做的模块收敛。

## 交付检查

- 只要改了结构、职责、命名或边界，就同步更新对应测试、文档和 skill
- 只要改了 Python 代码，就必须跑全量测试
- 如果测试失败，必须先修复再交付，或者明确说明阻塞原因
- 如果改动影响资源、IAM、环境变量或流程，必须同步更新文档
- 如果改动触及项目规则或 skill 边界，必须同步更新对应 skill 与 `AGENTS.md`
- 在提交或推送任何变更之前，先核查对应 PR 的真实状态；如果目标 PR 已经 merged 或 closed，就不得继续沿用同一条分支和同一份 PR 叙事，必须新开分支并重新建 PR
