# 架构重置式重构

当用户明确接受“无需考虑旧项目兼容性，可以大改代码”时使用本 skill。
如果用户仍然要求平滑迁移、兼容旧接口或控制改动面，改用 `project-delivery-guardrails`，不要误触这个 skill。

## 先判断是否值得重构

优先触发这个 skill 的信号：

- 当前目录、模块或调用链已经明显违背职责边界。
- 现有抽象只是历史兼容层，继续沿用只会让代码更乱。
- 文件落位错误到足以影响后续开发效率。
- 用户明确说可以不兼容旧结构，要求按当前最优形态重做。

## 重构前先输出目标结构

在真正改代码前先明确：

- 现有结构哪里错了，为什么错。
- 重构后的模块边界、目录落位和依赖方向是什么。
- 哪些文件应该删除、移动、合并、拆分。
- 哪些兼容层、过渡适配器、历史别名应直接去掉。
- 哪些测试、文档、部署入口必须一起调整。

目标结构必须是当前最优结构，不要为了照顾旧调用方式故意保留坏边界。

## 默认重构策略

- 优先删除无价值的兼容层、转发层、别名层和一次性胶水代码。
- 优先把代码移动到真正拥有职责的模块，而不是保留“旧路径 + 新实现”的双轨结构。
- 能合并的概念就合并，能消除的状态就消除，能直化的数据流就直化。
- 批量处理直接相关文件，不要把明显应该一起收敛的结构拆成多轮小修小补。
- 如果旧测试只验证过时结构，就同步重写测试，不为旧行为背书。

## 执行要求

1. 先给出重构边界，再动代码。
2. 相关目录、命名、接口、测试、文档一次性收敛。
3. 删除旧文件时同步清理引用、导出、脚本入口和说明文档。
4. 不保留“以后再删”的过渡代码，除非用户明确要求阶段性迁移。
5. 最终结果必须比原结构更简单、更可解释、更容易继续开发。

## 风险控制

- 虽然可以不考虑旧兼容性，但仍要先确认不会误删当前任务范围之外的有效功能。
- 任何大改都要跑仓库要求的全量验证；如果环境不足，明确说明未验证项。
- 如果重构改变了仓库规则、交付流程或 skill 边界，同步更新 `AGENTS.md` 和相关 skill。

## 交付口径

- 在结果说明里明确写出这次重构不保证旧兼容性。
- 清楚列出删除、移动和重命名的关键路径。
- 说明新的主入口、模块边界和后续开发应该遵循的结构。
