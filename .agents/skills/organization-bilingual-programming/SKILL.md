# 双语写作技能

## 目的

为当前组织级共享仓库提供统一的中英文写作规范。适用对象包括：

- `AGENTS.md`
- `.github/ISSUE_TEMPLATE/` 和 `.github/pull_request_template.md`
- `skills-src/organization/` 下的技能说明
- 脚本注释、README、提交信息和示例代码

这个仓库的内容以“模板、同步、worktree、GitHub 流程、生成视图”为主，因此双语规范的重点不是炫技式注释，而是让文案稳定、简洁、可复用、易同步。

## 核心原则

1. **中文叙述为主，英文标识保留原样**
   - 解释性文字优先中文
   - 文件名、命令、API 名、issue / PR 关键字、标签名保留英文原文
2. **避免混排歧义**
   - 不要在同一句里把中英文压成一团
   - 标题、列表项和表格字段要尽量保持同一语言风格
3. **模板语义优先**
   - issue / PR 模板里的字段名、占位符和关闭关键词要保持一致
   - 如果文案里出现 `Closes #123`、`Fixes #123`、`Resolves #123`，必须原样保留
4. **UTF-8 编码**
   - 所有中文文本文件使用 UTF-8 无 BOM
5. **术语统一**
   - 同一个概念在整个仓库里只用一套叫法
   - 例如“权威源 / 源目录 / 生成视图 / 下游视图 / 同步脚本”不要互相替换成多个近义词
6. **示例要贴合仓库背景**
   - 示例优先选择 issue、PR、worktree、sync、template 这些语境
   - 只有在确实需要代码示例时，才引入编程语言参考

## 写作建议

- 标题尽量短，且让读者一眼看出它是“仓库规则”还是“操作说明”
- 模板正文优先用命令式短句，少写空泛形容词
- 需要强调不可做的事情时，直接写禁止项，不要绕句
- 如果同一段里既有中文说明又有英文命令，先写中文解释，再给原生命令
- 术语第一次出现时可以中英并列，后文只保留一套主术语

```
│   ├── 01a-unreal-engine.md

│   ├── 04-go.md
│   ├── 05-java.md
│   └── 06-rust.md
└── scripts/
    └── validate-encoding.py
```

这些参考文件仍然保留语言级代码示例，但在这个仓库里，优先把它们当作“代码片段写作风格参考”，而不是仓库核心任务本身。

## 质量检查清单

- [ ] 中文叙述清楚，英文标识保留原样
- [ ] issue / PR / worktree / sync 等术语全仓库一致
- [ ] 模板字段、标签名和关闭关键词没有被改写
- [ ] 文件名、命令、路径、分支名没有被翻译成中文
- [ ] 文案中没有混乱的中英拼接句
- [ ] 所有中文文本文件使用 UTF-8 无 BOM
- [ ] 如果包含代码示例，示例本身符合对应语言惯例
